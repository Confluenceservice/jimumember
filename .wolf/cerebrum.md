## User Preferences

- **Python environments:** Use `uv` for installation and management (not pip)
- **Resume links:** Prefer surfacing resume links inline in the page (copyable), not only via email — applicants miss emails, and an in-page link is the safer fallback.
- **Multi-step user flows:** When a flow can race (uploads, form autosave), show a clear in-page reminder to the user, and disable the conflicting controls. Don't rely on server-side rejection alone.
- **Default branch is `main`:** Commit directly to `main` for general work. The `fix/*` branches are short-lived per-PR; once a `fix/*` branch's work is squash-merged into `main`, do NOT keep committing to that branch — new work goes to `main`.

## Key Learnings

- **Project:** eldaa-membership-checkout
- **Description:** Astro SSR app with:
- **Per-applicant serialisation pattern:** Both `apply.ts` (form save) and `upload-file.ts` (file upload) use a `Map<string, Promise<void>>` keyed by applicantId to chain writes for the same applicant. Mirror this pattern in any new endpoint that does a read-modify-write on a single applicant (e.g. doc counts, payment state).
- **Upload flow in `src/pages/professional/apply.astro`:** Step 8 has 7 categories (6 required + insurance optional). Each category has its own upload button + file input. `renderDocuments()` rebuilds the whole list on every state change — handlers must be idempotent and the in-flight lock must be re-applied after each re-render.
- **Resume link email:** `sendResumeLink` is sent once on registration (first save in `apply.ts`). The `/api/professional/resend-link` endpoint lets applicants re-fire it on demand.
- **Email verification gate (2026-06-12):** Professional applicant registration is now gated by the resume-link email. The no-token `POST /api/professional/apply` never returns the resume link in the response body — it returns `{ success: true, requiresVerification: true, emailSent }` instead. New rows write `email_verified = "FALSE"` to column AU; the GET handler (`?token=...`) flips it to `TRUE` on load (best-effort write). Legacy blank AU rows are treated as verified so existing applicants aren't locked out. The token-bearing POST keeps its current update + resume-link-return behavior since the token is itself proof of email control.
- **Two payment endpoints, split by plan:** Pro uses `/api/create-professional-checkout` (success → `/success`, cancel → `/professional/cancel`). Associate uses `/api/create-checkout-session` with `plan: "associate"` (success → `/associate-membership`, cancel → `/cancel`). Pick the right one per page — they have different success/cancel URLs and Pro-only fields.
- **Checkout buttons in this app use `bg-blue-600 hover:bg-blue-700`** (Tailwind), not the `var(--color-accent)` CSS theme var. Match the existing pages.
- **`src/pages/professional.astro` is a 4-line meta-refresh redirect to `/professional/apply/`** with a dead form (lines 8-181) still present from before commit 850a1d4. The meta refresh wins. Don't try to "fix" the file — it's preserved by user request as the production path.
- **Mailgun domain-level API keys return 404 on `GET /v3/{domain}`.** The recommended least-privilege scope is a per-domain send-only key. That key works for `POST /v3/{domain}/messages` but is forbidden from reading domain metadata. The `eldaa-health-alert` health probe (`checkMailgun` in `src/pages/api/health.ts`) used to do this GET and 404'd in production with a real key. The fix is to drop the network call and check env presence only — the strongest signal available without a key-scope roundtrip. Real send failures are still logged via `email-sender.ts`'s catch block to the audit sheet + Fly logger, so the email path isn't observability-blind.
- **Mailgun deliverability for new domains:** authentication headers (DKIM/SPF/DMARC all passing) do NOT prevent first-sends from a new domain going to Junk. Apple + iCloud + Proofpoint content classifiers weight sender reputation heavily; first-time senders to any address will get flagged. Use a recognisable From name (not "No Reply"), add `List-Unsubscribe` per RFC 8058 as a positive signal, and ask the first recipient per address to mark "Not Junk" + add to contacts. Volume of legitimate sends over days/weeks is the durable fix. There is no template change that defeats this on day 1.

## Do-Not-Repeat

- [2026-06-09] Do NOT enable concurrent uploads from the client. Even with a per-applicant server queue, the client must disable every upload/delete button while any upload is in flight — otherwise `renderDocuments()` can re-render fresh enabled buttons while another upload is still streaming, and the user can fire a second upload before the first one is counted. The `uploadsInFlight` Set + `refreshUploadLockUi` pattern in `apply.astro` is the canonical fix.
- [2026-06-09] Do NOT return `{ error }` from `resend-link.ts` (or similar email endpoints) without also returning the resume link on failure — if the email fails, the client should still be able to show the link to the user.
- [2026-06-10] Do NOT assume the working branch is whatever the session-start context says. The branch can switch underneath the session (background tasks, hooks, terminal multiplexing). Before any `git commit`, run `git branch --show-current && git status` to confirm where you are. If the branch has been merged into main via squash, commit to main directly — don't keep stacking commits on the merged branch. This wasted two re-deploys in the bug-047 session.
- [2026-06-12] Do NOT pass applicant-supplied email into a raw RFC822 `To:` line without first rejecting CR/LF. `email-sender.ts` interpolates the value into headers joined by `\r\n`, so a payload like `attacker@x.com\r\nBcc: spy@y.com` injects an arbitrary header. The `isValidEmail` helper in `src/pages/api/professional/apply.ts` rejects any value containing `\r` or `\n` and a malformed address before the row is ever created — apply the same guard to any future endpoint that builds a raw `To:` line.
- [2026-06-13] Do NOT use `GET https://api.mailgun.net/v3/{domain}` for a health probe if a domain-level API key (per-domain send-only, the recommended least-privilege scope) is in play. It returns 404 even when the key works for `POST /v3/{domain}/messages`. The probe was producing false `email: disconnected` Slack alerts in the workspace. For health probes with scoped keys, check env presence only — or use an account-level key if a real roundtrip is required (broader scope, broader access).
- [2026-06-14] Do NOT assume the DWD impersonation user from the runbook is still valid. `no-reply@eldaa.org.nz` was disabled when the Gmail→Mailgun migration dropped the only use of the Workspace account. If uploads start 500-ing with `UPLOAD_INTERNAL_ERROR`, the first thing to check is `fly secrets list` for `GOOGLE_WORKSPACE_IMPERSONATE_USER` and `fly logs --no-tail | grep google_service_account_auth` to see if `impersonating:false` (env unset) or impersonation 401s (env set to a dead user). The catch block in `upload-file.ts` returns generic 500 for any error including `storageQuotaExceeded` and `unauthorized_client` — so the response code alone is not enough to diagnose.
- [2026-06-14] Do NOT trust the Drive folder structure recorded in cerebrum without re-probing. PM/AM Applications were a direct child of the shared drive on 2026-06-13 and a child of "IT projects" on 2026-06-14. Workspace IT restructures can move folders without notice. `ensureFolderExists` is idempotent on name — if the apps folder ID is wrong, the upload code silently creates a new empty `PM Applications` at the wrong parent and orphans the existing data. Re-run `.run/probe-find-pm-folder.mjs` to verify parent before assuming a layout.
- [2026-06-14] Do NOT store hardcoded Stripe price IDs across envs when prices change. `STRIPE_PRICE_PROFESSIONAL` and `STRIPE_PRICE_ASSOCIATE` on Fly went stale when the product prices were updated in Stripe (\$120→\$150, \$50→\$75). The webhook then errors with "The price specified is inactive" and silently skips `markApplicantPaid` + `createApplicationReviewDoc` + confirmation email — the customer is charged but no side effects run. Production was updated to new active prices but staging was left pointing at archived ones. **Better pattern (TODO): use `lookup_key` in Stripe (e.g. `professional_annual_nzd`) and store the **product ID** in Fly env, not the price ID. Resolve the active price at boot via `stripe.prices.list({ product, active: true, lookup_keys: [...] })` and cache in memory. Price changes then require only a Stripe-side update, no env var, no redeploy. Add a `/api/health` `prices` field that asserts the cached prices are still `active: true` so silent deactivations surface as Slack alerts instead of failed webhooks.
- [2026-06-13] Do NOT trust Drive folder IDs across a Drive move. Items that are "moved" can keep their ID (and the code still works) or be "recreated" (new ID, old ID 404s or returns trashed/renamed content). Always probe live IDs against the Drive API after any user-reported Drive restructure, including checking for `trashed: true` (Drive returns 200 for trashed items by default — only `supportsAllDrives` reads the trash state). Also probe staging/production Fly secrets, not just `.env` — the .env may have been updated and Fly may still hold the dead value.

## Decision Log

- [2026-06-09] **Upload race fix uses both client lock + server queue.** Client lock prevents the bad UX of two buttons showing "Uploading..." and the in-flight "Uploading... N%" state being clobbered by `renderDocuments()`. Server queue closes the data-consistency gap (doc counts) in case any path bypasses the client (e.g. retry from another tab, programmatic client).
- [2026-06-09] **Resume link UX: inline copyable link + on-demand email button, not auto-resend.** Avoids duplicate emails on every save. The in-page link is always visible after Step 1; the email button is opt-in for users who want a fresh email.
- [2026-06-09] **`/renew/pro` and `/renew/associate` restore the payment-only flow** that commit 850a1d4 (May 14) removed when it turned `/professional` into a meta-refresh redirect. These are independent pages (not sub-paths of `/professional/apply/`) so the full application form is bypassed. URL params `firstName`/`lastName`/`phone`/`email` pre-fill the form. No application storage, no resume token — pure 4-input payment → Stripe Checkout.
- [2026-06-10] **`/renew/*` pages are WRONG for renewals** — they re-use the same `calcFirstTermAmount` prorata logic as new applications, but renewing existing members should pay the full price, not a prorated first term. **Use Stripe-hosted Payment Links for renewals instead.** Pro: `https://buy.stripe.com/5kQ7sKfOtaty397aVuak002`. Associate: `https://buy.stripe.com/9B600i31H0SYgZX3t2ak003`. Payment Links are fixed-price hosted checkouts with no form and no proration logic to maintain.
- [2026-06-10] **Health check + Slack alerting architecture.** Fly.io `[[http_service.checks]]` only routes traffic — it has NO native alerting on failure (no webhooks, no email). To get Slack alerts on health check failures, the pattern is: Fly's check pings the app's `/api/health` (the app probes Stripe + Gmail OAuth refresh token round-trip — see `src/pages/api/health.ts`), and a separate scheduler (Cloudflare Worker + GitHub Actions cron, since the CF account is on the free tier with no Workers Cron Triggers) pings `/api/health` periodically and posts to Slack on failure. Worker: `eldaa-health-alert` at `eldaa-health-alert.thomas-b02.workers.dev` (account `b029bfce2d4efabbcb5f8afd8028a913`). OAuth refresh tokens for Google don't expire on a schedule — they last until revoked (the `invalid_rapt` failure mode). Health probing is the right defensive layer, not scheduled rotation.
- [2026-06-11] **Gmail OAuth project is `mystical-runway-496019-r8`, not `stripe-billing-491503` as the runbook claims.** The runbook (`docs/runbooks/gmail-oauth-token-rotation.md`) has a stale project reference. The real client is `client_secret_531470041058-do9qba7t3c2osmrrri5i0dlhe9nk819u.apps.googleusercontent.com.json` (Downloads). The gcloud project `stripe-billing-491503` is the billing project for Stripe but not the OAuth client owner. If you `gcloud config set project stripe-billing-491503` and try to list OAuth clients via the IAM API you get `{}` — they live in a different project. Update the runbook next time someone touches it.
- [2026-06-11] **`fly ssh exec` is broken on stopped machines, and the secret values aren't returned by `fly secrets list` (truncated to a fixed-width digest).** To get the actual `GMAIL_OAUTH_*` env values from a Fly app you must: (1) `fly machine start <id>` to bring it back up, (2) wait for `started` state, (3) `fly ssh console -a <app> -C 'sh -c "printenv | grep GMAIL"'`. The console subcommand connects to a random started machine and returns stdout. Works for any secret — much faster than recreating the OAuth flow to "discover" the deployed client.
- [2026-06-12] **Magic-link email as a verification gate, not a separate OTP.** The proposed alternative (on-page OTP code) adds storage, expiry, resend, attempt-limiting, and a verify endpoint. Reusing the existing resume-link email as the verification token is simpler: clicking `?token=...` is itself the proof of control, the GET handler flips `email_verified` to `TRUE`, and the existing token capability model stays intact. Don't add an OTP layer unless UX research shows the email-link round-trip is too slow for users.
- [2026-06-12] **Legacy data migration via "blank means the old default"** in `parseEmailVerified`. Adding `email_verified` to a live sheet without backfilling is safe as long as the read path treats blank as the previously-implicit value (verified). Avoids a one-shot migration script, deploy-order coupling, and the risk of orphaning existing applicants.
- [2026-06-13] **Migrate transactional email from Gmail OAuth to Mailgun HTTP API.** The previous Gmail path was subject to Workspace Cloud session-control reauth every ~24h (surfaced as `invalid_rapt` on the OAuth refresh token), which made `/api/health` flap between `email: connected` and `email: disconnected` on a cron cycle and required manual token rotation via the runbook. Mailgun's API key is stable (no rotation cadence) and the new `checkMailgun` probe is env-var-only. Net: one less flaky alert source, no more 4-step "rotate the OAuth token" procedure for the user. The Drive/Docs/Sheets OAuth path stays on the service account + DWD impersonation (separate from the email path).
- [2026-06-13] **Drive folder IDs are fragile across Drive moves; shared drive IDs work as folder parents.** After a Workspace restructure, 8/22 of our top-level IDs went 404 or got trashed+renamed ("IT projects"). The fix: point `GOOGLE_DRIVE_APPLICATIONS_FOLDER_ID` at the **shared drive itself** (`0AO2UdklXEqS7Uk9PVA`), not at a leaf folder. `ensureFolderExists(sharedDrive, "PM Applications")` then finds the existing `1Nn-F5Cf-0xj02AYZFOZ3ba61QCviuyib` leaf (which SA = writer on) instead of trying to create a redundant nested copy. The same trick works for review docs — leave `GOOGLE_DRIVE_REVIEW_DOCS_FOLDER_ID` unset and let `google-docs.ts` fall back to the shared-drive root. SA-only access (no DWD impersonation) is fine: writer role on the leaf folders is enough for the upload code, and the spreadsheet ID is shared with the SA directly (SA can't see it in `files.list` but can `sheets.spreadsheets.get()` it). When probing, also unset the dead `REVIEW_DOCS_FOLDER_ID` from staging — `.env` changes don't propagate to Fly secrets.
- [2026-06-13] **Staging/production share a Drive after a Workspace restructure; isolate via a configurable folder-name prefix.** Once both envs point at the same shared drive, uploads from staging would land in the same `PM Applications` / `AM Applications` leaves as production. The fix is `getStagingPrefix()` (in `src/lib/staging.ts`) which reads `STAGING_PREFIX` from env and prepends it to top-level application subfolder names. Set `STAGING_PREFIX=testing-` on staging Fly only; production + local dev leave it unset (returns "" → no prefix, reuses existing folders). Applied to both folder creation (`PM Applications`, `AM Applications`) and index doc titles in `google-docs.ts`. `ensureFolderExists` is idempotent, so the first staging upload creates `testing-PM Applications` and `testing-AM Applications` as new siblings of the production ones — no data migration, full isolation from then on.
